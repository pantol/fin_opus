# Day 1 — simulated day in the life (2026-07-15)

First end-to-end walkthrough of the daily paper-trading loop, run on the
actual codebase with real GPW data. Everything below is **paper trading**
(no real orders), **zero LLM** (baseline `trend_momentum` v1 strategy), and
the multi-evening simulation ran in a **sandbox copy** of the database — the
real paper track record has NOT been started yet.

## How a production day works

| Time (Warsaw) | What happens |
|---|---|
| during the day | `make collect-loop` gathers ESPI/EBI filings + news into the DB (standalone, zero LLM). No intraday decisions — this is an EOD system. |
| 09:00–17:00 | GPW session. Yesterday's PENDING orders are conceptually filling at today's open; the system can only compute that after the data publishes. |
| ~19:00 | GPW publishes the session file (EOD prices + volume). |
| 19:30 | Cron: `make signals` → ingest → data gates (provenance / staleness / session coverage / config hash) → settle yesterday's orders at today's open → corporate actions → mark-to-market + trailing ATR stops → decide on today's close → commit. |
| 19:31 | Polish alert cards flush to Telegram — only AFTER the money state commits; without a token they queue (at-least-once delivery). |

Timing contract: a signal decided on session T's close becomes a PENDING
order that fills at session T+1's open — buy at ask, sell at bid, plus
slippage and commission (38 bps, min 3 PLN per side; spread 20 bps; slippage
10 bps; fills capped at 10% of session volume). Identical to the backtest
engine — `tests/test_paper_parity.py` enforces byte-level parity.

   2026-07-14): 34 bars across 17 tickers. Today's session file (2026-07-15)
   was not yet published at run time (~17:45); the 19:30 cron slot exists for
   exactly this reason.
2. **Bug found and fixed** — the first-ever `signals` run crashed:
   `config/universe.yaml` now carries unquoted `listed_from:` dates, PyYAML
   parses them into `datetime.date`, and `paper.loop.config_hash` was the
   first code path to JSON-serialize the universe → `TypeError`. Fixed with
   `default=str` (the convention `validation.config_hash` already used) at
   the three config-serialization sites, plus a regression test pinning that
   date objects hash identically to their ISO strings. 308 tests green.
   Commit `461e3a6`.
3. **Real dry-run** — `python -m app.cli signals --dry-run` (computed, then
   rolled back): would bootstrap a 100 000 PLN book and queue 7 BUY signals
   on the 2026-07-14 close.
4. **Sandbox simulation** — two full evenings replayed against a throwaway
   copy of `data/gpw.db` (details below), capturing the exact Telegram card

5. **Telegram mockup** — chat-style HTML preview of all captured cards:

## Sandbox simulation

Setup: `data/gpw.db` copied to a scratchpad `sandbox.db`; each evening ran
`paper_loop.run_signals` via the real code path with `--session`-style
clamps (`2026-07-13`, then `2026-07-14`) and a capture `send_fn`.
Reproducible with the CLI (cards print to console in token-less dry-run):

```bash
cp data/gpw.db /tmp/sandbox.db
python -m app.cli --db /tmp/sandbox.db signals --session 2026-07-13
python -m app.cli --db /tmp/sandbox.db signals --session 2026-07-14
```
